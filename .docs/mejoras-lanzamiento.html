Mejoras en base al lanzamiento

NutriApp — Backlog post-lanzamiento inicial

Documento local · No subir a GitHub

Lista de mejoras identificadas a partir del feedback del lanzamiento inicial y observaciones propias. Ordenadas por área. Ninguna está implementada aún — son el roadmap inmediato.

1

Feedback negativo al agregar comidas

UX

Problema: al registrar una comida no queda claro si se guardó correctamente. No hay confirmación visual ni mensaje de éxito que indique al usuario que la acción funcionó.

  • Agregar toast/snackbar de confirmación al guardar una comida.
  • Mostrar estado de error si la comida no se pudo guardar.
  • Considerar animación de entrada en la lista al agregar un ítem nuevo.
2

Campos de edad y altura en el perfil

Feature

Propósito: con edad, altura y peso se pueden calcular automáticamente las calorías de mantenimiento (TDEE) y los macros recomendados, personalizando la experiencia sin que el usuario tenga que ingresar esos valores manualmente en el chat.

  • Agregar campos age (número) y height (cm) al modelo Profile en Prisma.
  • Mostrarlos en el formulario de creación y edición de perfil.
  • Incluirlos en el contexto del sistema del NutriChat para cálculos de TDEE.
3

Opción para cambiar el objetivo desde el perfil

Feature

Problema: el campo goal del perfil actualmente solo se puede definir al crear el perfil. El usuario no tiene forma de cambiarlo si su objetivo evoluciona.

  • Agregar un botón o sección "Editar objetivo" en la pantalla de perfil.
  • Mostrar un selector con las opciones disponibles (bajar de peso, mantener, ganar masa muscular, etc.).
  • Llamar al endpoint de actualización de perfil al guardar.
4

API de alimentos para Sudamérica y Argentina

Infraestructura

Problema: la API actual (USDA FDC) está orientada al mercado estadounidense. Muchos alimentos típicos de Argentina y Sudamérica no aparecen o aparecen con nombres incorrectos (milanesa, locro, mate, facturas, yerba, etc.).

  • Opción A: agregar una segunda API complementaria (ej. Open Food Facts, que tiene base de datos global y permite búsqueda en español) y hacer fallback si USDA no devuelve resultados relevantes.
  • Opción B: crear una base de datos propia de alimentos argentinos más comunes, curada manualmente.
  • Opción C: permitir al usuario agregar alimentos personalizados con sus propios macros.
  • Evaluar combinar A + C como solución más completa.
5

Validación de cuenta por email al registrarse

Auth

Propósito: verificar que el email ingresado es real antes de activar la cuenta. Mejora la calidad de la base de usuarios y permite recuperación de contraseña en el futuro.

  • Al registrarse, enviar un email con un token de verificación (link de confirmación).
  • El usuario no puede iniciar sesión hasta confirmar el email.
  • Requiere un servicio de envío de mails: Resend o SendGrid (ambos tienen plan gratuito).
  • Agregar campo emailVerified (boolean) al modelo User en Prisma.
  • Agregar endpoint GET /auth/verify?token=... en el backend.
→

Notas adicionales

  • Priorizar #1 (feedback visual) y #3 (cambiar objetivo) primero — son cambios de bajo esfuerzo y alto impacto.
  • #2 (edad/altura) y #5 (verificación por email) requieren cambios en la base de datos (migración Prisma).
  • #4 (API de alimentos) es el de mayor esfuerzo — evaluar Open Food Facts como complemento primero antes de construir algo propio.
